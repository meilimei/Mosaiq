## 1. 背景

v0.9 的 CHANGELOG 标题写得很自洽:「v0.9 CLI + Detection Lab polish」。但回头
看实际产物，有两个**结构性缺口**与产品长期叙事错位：

1. **零外部分发**: CLI（~4000 LOC，16 个 subcommand，64 个 vitest case）目前
   只能在 monorepo 内 `pnpm mosaiq`。外部用户无法 `npx mosaiq` /
   `npm i @mosaiq/sdk`。v0.9 的 reach 是 0。没有 npm 公开发行，CLI parity 这个
   工作量没法兑现成「Day 1 SDK + CLI 可用」的产品宣称（PRD §2 五大差异化第
   4 条 "真 Dev-First"）。
2. **零自动反检测回归 gate**: SDK 注入逻辑 (canvas / webgl / audio / WebRTC /
   UA-CH) 改一行，没有任何 CI 检查会捕捉「这个 patch 让 sannysoft webdriver
   检测从 0 hits 涨到 1 hits」之类的回归。v0.9 ship 了能做这件事的 CLI
   (`detection-lab run-all --fail-on-regression`，phase 9.10)，但**没接到
   `.github/workflows/`**。现状是: SDK contributor 改完发 PR，reviewer 只能凭
   代码 review 判断是否引入了反指纹回归。

两个缺口互为前提:
- 没有 CI gate 就敢公开发行 → 外部用户跑到 0.10.3 发现反检测退化，伤口在外
- 没有公开发行就只有内部用 CI gate → 投入 ROI 低，自己用自己 review

所以 v0.10 **双轨同推**: Distribution (10.1-10.5) + Self-Gating (10.6-10.9)。

### 1.1 v0.9 → v0.10 关键差异

| 维度 | v0.9 | v0.10 |
|---|---|---|
| 安装路径 | git clone + pnpm install | `npm i -g @mosaiq/cli` / `npm i @mosaiq/sdk` |
| 用户范围 | monorepo 内 contributor | 任何 Node.js 用户 |
| `@mosaiq/cli` 包状态 | `private: true` | 公开发行（`private` removed） |
| `@mosaiq/sdk` README | 不存在 | 完整 README + npm install / quickstart |
| Playwright patch 分发 | pnpm patchedDependencies (workspace-only) | npm publish 后通过 `patch-package` postinstall 复用 |
| 版本管理 | 手工 `chore(release): vX.Y.Z` commit | changesets + lock-step 三发包包 |
| CI 反检测 gate | 无 | `.github/workflows/detection-lab.yml` 在 PR + nightly 跑 |
| Baseline run | 手工 ad-hoc | `tests/fixtures/baseline-runs/<persona>/<run>.json` 提交版控 |

根 `package.json`（`name: "mosaiq"`）**没有** `"private": true`，即使从根目录
意外 `pnpm publish` 也能发出去。10.1 顺手补上。

### 2.2 Playwright 补丁现状（v0.10 最大工程难点）

`pnpm.patchedDependencies` 在 root `package.json` 里注册了
`playwright-core@1.59.1` 的 patch（`patches/playwright-core@1.59.1.patch`，
302 行）。补丁实质: 集成 [rebrowser-patches](https://github.com/rebrowser/rebrowser-patches)
的 `__re__emitExecutionContext` 等钩子，修复 Playwright 的 `Runtime.enable`
检测向量（CDP 自动暴露 execution context，反检测站如 creepjs / sannysoft 据此
判定「这是自动化」）。没有这个 patch，SDK 在 sannysoft 上至少多 1 个
high-severity hit（`webdriver` surface）。

**pnpm patchedDependencies 只在 monorepo 内有效** — `workspace + pnpm install`
时 pnpm 把 patch 作用到 `node_modules/playwright-core/`。一旦
`npm i @mosaiq/sdk` 安装到外部用户的 npm/yarn/pnpm 项目，pnpm
patchedDependencies 不会生效（甚至不会被 npm/yarn 读取），用户的
playwright-core 是裸的。

v0.10 必须解决这个分发缺口，方案见 [§5 Phase 10.2](#5-phase-102--playwright-补丁分发策略-patch-package)。

### 2.3 现有 CI (ci.yml)

`.github/workflows/ci.yml` 在 v0.9 phase 9.10 已加固:
- typecheck × 4 包（persona-schema / sdk / cli / desktop）
- vitest × 4 包（persona-schema 26 / sdk 593 / cli 64 / desktop 45，合计 728 tests）
- Phase 7.0 captured-profiles drift check

这是「代码层」守卫，不跑 Chromium，不发 detection 请求。v0.10 的
detection-lab.yml 是另一半 —「行为层」守卫，跑真 Chromium 真请求真打分。

### 2.4 LICENSE 草案状态

根目录 `LICENSE.md` 标注「状态: 草案，最终方案在公司主体注册完成后由法务确认」。
README + WHITEPAPER 多处提及「Persona Schema、CLI、TypeScript SDK 计划以
Apache 2.0 开源」，但当前 `LICENSE.md` 内容是策略说明，不是正式的 Apache-2.0
license text。10.1 配套要为三个发包包写真正的 Apache-2.0 LICENSE 文件
（`packages/<pkg>/LICENSE`），与 npm registry 显示页面一致。

## 5. Phase 10.2 — Playwright 补丁分发策略 (patch-package)

### 5.1 决策矩阵

| 方案 | 实现 | 优点 | 缺点 | v0.10 选择 |
|---|---|---|---|---|
| A. `patch-package` postinstall | sdk 依赖 patch-package，patches/ 随包发，postinstall 自动 patch 用户的 playwright-core | 标准做法（antidetect 社区主流）；patch 透明可审；维护成本低 | 依赖 postinstall (CI 关掉就废)；patch 会改用户其他代码引用的 playwright-core | **★ 推荐** |
| B. 自己 fork `@mosaiq/playwright-core` | 发布 vendored patched playwright-core 为 `@mosaiq/playwright-core@1.59.1`，sdk 改 import 到这个 fork | 用户 0 配置；不污染用户原始 playwright-core | 维护负担大（每次 playwright 升级要重 patch + 发新版）；包体积大 | v0.11+ 考虑 |
| C. Runtime monkey-patch | sdk import 时 patch playwright-core 导出的 class 方法 | 0 install-time 副作用 | brittle（playwright internal 改名就废）；patch 内容 302 行手搓 JS 不现实 | 不选 |
| D. Land upstream | 提 PR 到 Playwright | 一劳永逸 | Playwright 官方对 stealth/antidetect patch 立场负面（security model 冲突），90% 被拒 | 不选 |

### 5.2 方案 A 实施

**1. 把 `patches/playwright-core@1.59.1.patch` 复制到 `packages/sdk/patches/`**

pnpm 的 patchedDependencies 在 monorepo 内沿用根 `patches/`，但 npm publish
时只有 `packages/sdk/` 目录里的文件被打进 tarball。所以需要复制（不是符号链接，
否则 npm publish 后丢失）。两份文件用 CI drift check 锁定一致（10.2 顺手加）。

**2. `packages/sdk/package.json`**:

```json
{
  "dependencies": {
    "@mosaiq/persona-schema": "workspace:*",
    "https-proxy-agent": "^7.0.6",
    "patch-package": "^8.0.0",
    "playwright-core": "1.59.1",
    "socks-proxy-agent": "^8.0.5"
  },
  "scripts": {
    "postinstall": "patch-package --patch-dir patches",
    ...
  },
  "files": ["dist", "patches", "README.md", "LICENSE"]
}
```

**3. drift check** (`scripts/check-sdk-patch-drift.ts`):
- 比对 root `patches/playwright-core@1.59.1.patch` 与
  `packages/sdk/patches/playwright-core@1.59.1.patch` 是否字节一致
- 在 `.github/workflows/ci.yml` 加一步 `pnpm run check-sdk-patch-drift`
- 失败给出明确修复指令: `cp patches/playwright-core@1.59.1.patch
  packages/sdk/patches/`

**4. 文档**:
- sdk README 加「Why postinstall? Playwright 在 v0.10 之前需要一个 302-line
  rebrowser-patches patch 来关掉 Runtime.enable 检测向量。该 patch 必须作用到
  你 node_modules 里的 playwright-core 上。所以 `npm i @mosaiq/sdk` 会自动跑
  `patch-package` 来 apply。如果你的 CI 关闭了 postinstall（罕见），需要
  手动 `npx patch-package --patch-dir node_modules/@mosaiq/sdk/patches`」
- changelog 单独一段强调首次安装会执行 postinstall

### 5.3 风险

| 风险 | 影响 | 缓解 |
|---|---|---|
| 用户 CI 关闭 postinstall（如 `npm ci --ignore-scripts`） | playwright-core 不 patched，sannysoft `webdriver` hit | sdk runtime startup 检查是否被 patched (`typeof CRSession.prototype.__re__emitExecutionContext === 'function'`)，否则 warn |
| 用户已有自己的 playwright-core patch | 冲突 | patch-package 报 conflict 时已经够明确；README 说明 |
| Playwright 升级（1.59.1 → 1.60.x） | patch 失效 | 锁 `playwright-core: "1.59.1"`（精确版本，无 caret）；升级 playwright 是独立工程，明确写在 v0.11 候选 |

### 5.4 验证

- 在临时目录 `mkdir /tmp/sdk-test && cd /tmp/sdk-test && npm init -y && npm i
  @mosaiq/sdk@latest`，看 postinstall log 包含 `Applying patches...
  playwright-core@1.59.1`，然后跑一段最小代码确认 webdriver detection 被关
- 在带 `--ignore-scripts` 的安装路径下，sdk warn 一行 stderr 提示

### 5.5 估时

2-3h（patches 复制 + drift script + sdk README 段落 + smoke test）。

## 6. Phase 10.3 — Build pipeline + .npmignore + tarball 审计

### 6.1 目标

- 三个发包包都有可靠 `prepublishOnly` hook
- 每个包都有 `.npmignore` 或精确的 `files` 字段
- 已发 tarball 不包含 bench / chromium-fork / 测试源码 / patches stub paths

### 6.2 实施详情

**Per-package `prepublishOnly` script**:

```json
{
  "scripts": {
    "prepublishOnly": "pnpm run clean && pnpm run build && pnpm run typecheck"
  }
}
```

防御性: 即使 changesets 自动 publish CI 跑漏了 build 步骤，prepublishOnly
也会兜底；同时本地 `pnpm publish` 也安全。

**`packages/sdk/.npmignore`** (或者 negative pattern in `files`，二选一；
我建议 explicit `files` 列表):

```json
{
  "files": [
    "dist",
    "patches",
    "README.md",
    "LICENSE"
  ]
}
```

这样 `bench/` / `src/` (sources) / `examples/` / `tsconfig*.json` /
`vitest.config.ts` 都不会进 tarball。

**关键审计点**: SDK `bench/` 目录里有 captured-profiles JSON 大文件（数 MB），
绝对不能进 npm tarball。`files: ["dist", "patches", ...]` 默认就把它排除了。

**`packages/cli/.npmignore`** (同样，用 `files`):

```json
{ "files": ["dist", "bin", "README.md", "LICENSE"] }
```

CLI 的 `bin/mosaiq.js` 是入口 shim（5 行 spawn dist），所以 bin 必须包含。

**`packages/persona-schema/.npmignore`**:

```json
{ "files": ["dist", "README.md", "LICENSE"] }
```

**Tarball audit script** `scripts/audit-tarballs.ts`:
- 对三包跑 `pnpm pack`，拿到三个 `.tgz`
- 用 `tar -tzf` 列文件清单
- assert: 不含 `*.test.{ts,js}` / `bench/` / `examples/` / `chromium-fork`
  / `node_modules` / `.git`
- assert: 含 `package.json` / `README.md` / `LICENSE` / `dist/**/*.js` /
  `dist/**/*.d.ts`
- 失败给出哪个包哪些文件越界

CI 加一步 `pnpm run audit-tarballs`，发版前必跑。

### 6.3 验证

- 三包 `pnpm pack --dry-run` 输出 file list 由 audit script 验证
- 实际 tarball 大小预期: persona-schema ~30 KB, sdk ~500 KB (含 patches),
  cli ~80 KB

### 6.4 估时

2-3h（audit script + tarball 手工 inspect）。

## 7. Phase 10.4 — Changesets 引入 + lock-step 配置

### 7.1 目标

- 引入 [changesets](https://github.com/changesets/changesets) 做版本管理
- 三发包包配置成 fixed group（lock-step bump，与 v0.9 实际节奏一致）
- 集成 GitHub Action `changesets/action` 自动开 "Version Packages" PR
- CHANGELOG 兼容现有手写格式

### 7.2 实施详情

**安装**:

```bash
pnpm add -Dw @changesets/cli
pnpm exec changeset init
```

产物: `.changeset/config.json` + `.changeset/README.md`。

**`.changeset/config.json`**:

```json
{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",
  "changelog": ["@changesets/changelog-github", { "repo": "meilimei/Mosaiq" }],
  "commit": false,
  "fixed": [
    ["@mosaiq/persona-schema", "@mosaiq/sdk", "@mosaiq/cli"]
  ],
  "linked": [],
  "access": "public",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": ["@mosaiq/desktop"]
}
```

关键决策:
- **`fixed: [["@mosaiq/persona-schema", "@mosaiq/sdk", "@mosaiq/cli"]]`** —
  三包同进同退（v0.9 实际节奏就是这样）。任何一个改 minor，三个一起 minor。
- **`ignore: ["@mosaiq/desktop"]`** — desktop 不发 npm，不参与 changeset 计票
  （desktop 内部仍然手工 bump 与三发包包同步，或在 release.yml 里跟随
  bump）。
- **`changelog: @changesets/changelog-github`** — 自动从 PR 标题 / 作者
  生成。但项目现有 CHANGELOG.md 是手写中文长文，转 changesets-github 会丢失
  这个风格。**折中**: 仍保留 `CHANGELOG.md` 手写（在 chore(release) commit
  里手动 append），changesets 只管 version bump 和 npm publish；
  `.changeset/*.md` 文件给 reviewer 看 PR 期间的 intent，merge 后 changesets
  bot 自动转换。

**`.github/workflows/release.yml`** 新增:

```yaml
name: Release
on:
  push:
    branches: [main]
concurrency: ${{ github.workflow }}-${{ github.ref }}
jobs:
  release:
```

### 7.3 现有手工 release 流程的兼容

## 9. Phase 10.6 — CI fixture personas + baseline runs（committed）

### 9.1 目标

- 把 1-2 个 canonical persona 作为 fixture 提交到仓库 (`tests/fixtures/personas/`)
- 把对应的 baseline detection run 也提交 (`tests/fixtures/baseline-runs/`)
- 提供脚本 `scripts/refresh-baseline.ts` 让维护者本地 / nightly cron 刷新
  baseline

### 9.2 实施详情

**Fixture personas** (`tests/fixtures/personas/`):

```
tests/fixtures/personas/
  win11-chrome-us.json         # 基础 Win11 Chrome US locale
  macos-chrome-us.json          # macOS Chrome US locale  (可选, MVP 只一个)
```

- 这两个 persona 用 `mosaiq personas export --include-secrets=false`（默认）
  生成，proxy.password 等 secrets 已被 redact
- 命名: 不用 `<template>-<random>` 而用稳定 id (`win11-chrome-us`)，方便
  baseline 文件按 persona id 寻址
- 文件大小: 每个 ~3-5 KB
- 写在 doc footer: 「不要手工编辑这两个文件；用 `pnpm refresh-baseline` 重生」

**Baseline runs** (`tests/fixtures/baseline-runs/`):

```
tests/fixtures/baseline-runs/
  win11-chrome-us/
    baseline.json             # 完整 DetectionRun JSON
    baseline.md               # formatDetectionRunMarkdown 输出（human-readable）
```

- baseline 来源: 在「干净」的 main HEAD 上, ubuntu-latest CI runner 环境跑出
  的 detection run（不是开发者本地跑的；本地跑的会有 hardware fingerprint
  差异）。所以**第一次的 baseline.json 是在 10.6 引入的 CI workflow 里
  artifact 下载下来手工 commit 的**，不是开发者本地生成
- baseline.md 是 baseline.json 的 markdown 投影，跑 `formatDetectionRunMarkdown
  (run, { includeMeta: false })` 生成（去掉 timestamp / runId 等不稳定字段，
  diff-friendly）
- 这两个文件是「golden file」，git diff 可读，PR review 能看出 baseline 改了
  哪些 surface

**Refresh script** `scripts/refresh-baseline.ts`:

```typescript
// pseudo
import { loadPersona, runDetection, formatDetectionRunMarkdown } from '@mosaiq/sdk';
import { writeFileSync } from 'fs';

const FIXTURES = ['win11-chrome-us'];

for (const id of FIXTURES) {
  const persona = loadPersonaFromFile(`tests/fixtures/personas/${id}.json`);
  const { raw, score } = await runDetection(persona, {
    artifactDir: undefined,  // 不写 screenshot, 减少 diff
    personaTemplate: id,
  });
  const run = buildRun(raw, score);  // 同 desktop main 的 buildCompletedRun
  // 但 strip 不稳定字段:
  delete run.runId;
  delete run.startedAt;
  delete run.finishedAt;
  for (const site of run.raw.sites) {
    delete site.startedAt;
    delete site.finishedAt;
    delete site.durationMs;
  }
  writeFileSync(
    `tests/fixtures/baseline-runs/${id}/baseline.json`,
    JSON.stringify(run, null, 2),
  );
  writeFileSync(
    `tests/fixtures/baseline-runs/${id}/baseline.md`,
    formatDetectionRunMarkdown(run, { includeMeta: false }),
  );
}
```

关键: **strip 时间戳 + durationMs**, 因为这些字段每次跑都不同，会污染 diff。
baseline 只保留 "stable behavior" 字段（hits / metrics / extracted）。

### 9.3 SDK / CLI 配套（可能需要的小改）

- v0.9 的 `compare` / `diffRuns` 默认拿两个完整 run JSON, 不接受 stripped
  baseline。需要确认 stripped baseline 仍能 round-trip 通过 `compare` 函数
  （`startedAt` 等字段是 optional 还是 required）
- 如果是 required，给 baseline.json 重新填 dummy timestamp（`"startedAt":
  "1970-01-01T00:00:00.000Z"`）作为占位，避免 SDK schema 校验失败
- 这一小步可能要 SDK 改 1 行（把 timestamp 字段标 optional in
  `DetectionRunSchema`），或者 baseline 不 strip（接受 diff noise）。视情况
  在 10.6 里决定

### 9.4 估时

2-3h（refresh script + 在 CI 里生成第一次 baseline + 手工 inspect 后 commit）。

## 10. Phase 10.7 — `.github/workflows/detection-lab.yml`（反检测回归 gate）

### 10.1 目标

- 新 workflow 在 PR + push:main + nightly cron + workflow_dispatch 跑
  detection-lab
- 用 fixture persona, 跑 baseline-runs 里 stored baseline 做比对
- Regression（任意 site 多出 hit / weightedHits 涨 / status flip ok→fail）
  → exit 1, workflow 红
- Pass → workflow 绿, status check `detection-lab / regression` ✅

### 10.2 实施详情

**`.github/workflows/detection-lab.yml`**:

```yaml
name: Detection Lab
on:
  pull_request:
    branches: [main]
    paths:
      - 'packages/sdk/**'
      - 'packages/persona-schema/**'
      - 'tests/fixtures/personas/**'
      - 'tests/fixtures/baseline-runs/**'
      - '.github/workflows/detection-lab.yml'
  push:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'    # 每周一 06:00 UTC nightly drift
  workflow_dispatch: {}

concurrency:
  group: detection-lab-${{ github.ref }}
  cancel-in-progress: true

jobs:
  detection:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    strategy:
      fail-fast: false
      matrix:
        persona: [win11-chrome-us]
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with: { version: 9.12.0 }
      - uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm --filter @mosaiq/persona-schema --filter @mosaiq/sdk --filter @mosaiq/cli build
      - name: Install Playwright Chromium
        run: pnpm --filter @mosaiq/sdk exec playwright install --with-deps chromium
      - name: Seed fixture persona
        run: pnpm mosaiq personas import tests/fixtures/personas/${{ matrix.persona }}.json --on-conflict overwrite
      - name: Run detection-lab
        id: run
        run: |
          pnpm mosaiq detection-lab run ${{ matrix.persona }} --json > candidate-run.json
          # candidate-run.json 全文; 后续 compare 用
      - name: Compare against baseline
        run: |
          # baseline 在 tests/fixtures/baseline-runs/<persona>/baseline.json
          # 把 candidate-run.json 写进 ~/.mosaiq/detection-runs/<persona>/<runId>.json
          # 不太行 (runId 是 nondeterministic). 改用 SDK diffRuns 直接对 JSON:
          node scripts/ci-compare-baseline.mjs \
            tests/fixtures/baseline-runs/${{ matrix.persona }}/baseline.json \
            candidate-run.json \
            --fail-on-regression
      - name: Upload artifacts on failure
        if: failure()
        uses: actions/upload-artifact@v4
        with:
          name: detection-lab-${{ matrix.persona }}-${{ github.sha }}
          path: |
            candidate-run.json
            ~/.mosaiq/detection-runs/${{ matrix.persona }}/**
          retention-days: 14
      - name: Generate markdown summary
        if: always()
        run: |
          pnpm mosaiq detection-lab show-run ${{ matrix.persona }} ... >> $GITHUB_STEP_SUMMARY
```

**`scripts/ci-compare-baseline.mjs`** (新增):

```javascript
// pure CJS-friendly wrapper around SDK diffRuns,
// 接两个 file path, exit 1 on regression
import { readFileSync } from 'node:fs';
import { diffRuns } from '@mosaiq/sdk';

const [, , baselineFile, candidateFile, ...flags] = process.argv;
const baseline = JSON.parse(readFileSync(baselineFile, 'utf-8'));
const candidate = JSON.parse(readFileSync(candidateFile, 'utf-8'));
const diff = diffRuns(baseline, candidate);

console.log(formatDiffMarkdown(diff));   // 写 stdout 给 GITHUB_STEP_SUMMARY

if (flags.includes('--fail-on-regression') && diff.isRegression) {
  process.exit(1);
}
```

注意: v0.9 phase 9.8 已经把 `diffRuns` / `RunDiff` 从 CLI 提到 SDK，所以
script 可以直接 `import` 它，逻辑零重复。

### 10.3 PR comment bot（可选 polish, 推到 10.9）

10.7 MVP: failure 时 workflow 红 + artifact 上传 + GITHUB_STEP_SUMMARY 显示
diff markdown。已经够用。

10.9 升级: 在 PR 上额外贴一条 bot comment, 用
`peter-evans/create-or-update-comment@v4`, 把 diff markdown 复制进去, 比
"点进 Actions 看 step summary" 体验好。

### 10.4 容量 / 性能预估

- ubuntu-latest runner: 4 CPU / 16 GB / SSD
- Playwright install chromium: ~30s（带缓存可 5s）
- 12 site detection run: 5-10 min (12 × 30-60s)
- 全 workflow: ~15 min。在 30 min timeout 内充裕
- 每月 GH Actions 公开 repo 免费分钟: 2000 min。每周 4 PR × 15 min = 60 min,
  +nightly 15 min = ~75 min/week, ~300 min/month, 远低于 quota
- 如果 PR 数量爆涨需要降本: paths 过滤已经限制了只在 SDK / fixture 改动时触发

### 10.5 容错策略

- 单 site 失败 (网络) 不能 fail workflow（噪声大）
- 用 `mosaiq detection-lab run --retries 3 --timeout 60000` 容忍
- 如果 12 站里 ≤2 站失败但其他都 OK 且 weightedHits 没涨 → pass
- 如果 ≥3 站失败 → fail（baseline 是 12 站都成功的，这种程度的失败可能是环境
  问题不是回归，但保守起见还是失败让 maintainer 看一眼）
- 这套阈值通过 `--fail-on-network-failure-threshold 2` (10.7 在 CLI 加的
  新 flag) 或 ci-compare-baseline.mjs 内部逻辑实现

### 10.6 估时

3-5h（workflow yaml + ci-compare-baseline script + 阈值微调 + 在 fork repo
里 dry run 验证一次完整通过）。

### 11.2 实施详情

**`.github/workflows/refresh-baseline.yml`**:

```yaml

  schedule:
    - cron: '0 4 * * 0'    # 每周日 04:00 UTC, nightly 之前
  workflow_dispatch: {}

jobs:
  refresh:
    runs-on: ubuntu-latest
    permissions:
      contents: write

      - uses: pnpm/action-setup@v4

        uses: peter-evans/create-pull-request@v6

          commit-message: "chore(baseline): refresh detection-lab baseline"

            ```
            <inserted by refresh-baseline.ts using diffRuns + format>

  缓解: `peter-evans/create-pull-request` 默认会 reuse 已存在 PR 不重开，
  4 次只有 1 个 PR 在飞

### 11.4 估时

2-3h（workflow + refresh script 加 "3-run consensus" 逻辑 + create-pull-request
配置）。

| R10 | GH Actions 每月 2000 min quota 在 PR 高峰超出 | CI 跑不了 | paths 过滤限制只在 SDK / fixture / workflow 改动触发；nightly 是固定 ~15 min/week 可控；最坏情况付费 |

## 14. 时间估算 & 节奏

| Phase | 任务 | 估时 | 累计 |
|---|---|---|---|
| 10.1 | per-package metadata + license + sdk README + 根 private | 2-3h | 3h |
| 10.2 | patch-package + sdk patches/ + drift check | 2-3h | 6h |
| 10.3 | prepublishOnly + .npmignore + tarball audit script | 2-3h | 9h |
| 10.4 | changesets init + lock-step config + release.yml | 3-4h | 13h |
| 10.5 | npm org reserve + dry-run + first publish + README/QUICKSTART 更新 + tag | 2-4h | 17h |
| 10.6 | fixture personas + baseline runs + refresh script | 2-3h | 20h |
| 10.7 | detection-lab.yml workflow + ci-compare-baseline.mjs | 3-5h | 25h |
| 10.8 | refresh-baseline.yml + 3-run consensus | 2-3h | 28h |
| 10.9 | PR comment bot | 1-2h | 30h |

**Total: 19-30h，拆 7-9 个 session 推**。每个 phase 独立可 commit，不必等下一
phase 完成。

**推荐顺序**: 10.1 → 10.2 → 10.3 → 10.4 → 10.5 → 10.6 → 10.7 → 10.8 → 10.9。
理由:
- 10.1-10.5 是 Track A（Distribution），10.6-10.9 是 Track B（Self-Gating）；
  两轨独立，但 publish 先于 gate 上线更合 v0.10 「先有外部用户、再有内部 CI
  保护」的叙事
- 10.5 真发布**必须**在 10.1-10.4 全完成之后（dry-run 全通过）才执行；这是
  整个 v0.10 的「不可逆」节点（npm 24h 撤回窗口）
- 10.6 在 10.5 之后做，因为 10.5 publish 之后 CI 里 `pnpm mosaiq` 用的是
  workspace 链接的 0.10.0；如果想用真 npm 0.10.0 也可以但意义不大
- 10.7 / 10.8 / 10.9 可以独立切并行，但 10.7 是 MVP，10.8 / 10.9 是 polish

也可以做「快慢拆分」: 10.1-10.5 一周内 ship 「v0.10.0 npm 首发」；10.6-10.9
后续作为 v0.10.1 / v0.10.2 patch 节奏 ship。但本规划倾向于**单次 v0.10.0
同时上 Distribution + Self-Gating**，对外宣发更整齐。
